How to read this

The win scales with how verbose the baseline is by default. Against Tailwind — a wrapper <div> plus a dozen re-emitted utility classes per element — MaxHTML cuts ~69%. Against Pico (classless; <article> is already a card) and Shoelace (one component tag per widget), both already markup-terse, the edge is ~27%.

That ~27% is real but concentrated at the app level, where MaxHTML's composite layouts (mh-app, mh-sidemenu, mh-page) replace hand-assembled shells that Pico and Shoelace have no component for. On atoms and simple content the edge shrinks to ~10–15% and can reverse — Pico's classless <article> beats an mh-card when there's no layout to arrange. Tokens alone don't justify the kit; consistency, app-shell coverage, no-JS rendering, and ejectability do.

Method & honesty notes

All four kits build the identical, content-locked screen (same stats, same rows, same dialog), so none inflates its count with extra content. Counted with tiktoken over the emitted body only — <!doctype>, <head>, the <link> stylesheet, and behavior scripts excluded for every kit. The MaxHTML side is verified clean: zero <div>, <span>, class=, style=, or <style>. cl100k_base gives the same picture.

Caveat: these are hand-authored equivalents, so absolute numbers are a first-order signal — one author wrote all sides, and the Pico/Shoelace idiom is more opinionated than Tailwind's, so their numbers are softer. The credible settler is model-generated: emit each screen four ways (each kit's default prompt vs. MaxHTML given PROMPT.md) in any chat, then count what the model actually emits. That removes authoring bias and doubles as a compliance test — no API required.

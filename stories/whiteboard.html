Whiteboard (recipe — escape hatch)

A diagram editor — mh-app shell + mh-canvas with a floating mh-toolbar, an SVG connector layer, and draggable mh-note stickies. This is the first recipe that leaves the rail, and it's here to show exactly where and how.

Live (drag the notes)

What's on-rails vs. the escape hatch

On-rails: the app shell, the floating mh-toolbar (buttons + aria-pressed), the note's visual box (paper, shadow, type — all in the stylesheet), and the dotted-grid canvas background. The toolbar's anchor (top-centre) is reusable chrome, so it lives in the sheet too.

Escape hatch: a canvas's substance is per-element position and connector geometry — that's data, unique per note, so it can't live in an external stylesheet. Positions ride on the custom properties --x/--y/--w (and colour on --note) set via style="…". The style attribute carries only data here — never a visual rule — but it is still an exception to rule 1 ("never write style="). Connectors are a raw <svg> of <line>s with inline coordinates.

<mh-canvas>
  <mh-toolbar>…tools…</mh-toolbar>
  <svg><line x1=252 y1=140 x2=340 y2=120 marker-end="url(#arrow)"></line>…</svg>
  <mh-note style="--x:60px; --y:80px; --note:#fde68a">
    <strong>Kickoff</strong><p>Align on goals & owners.</p>
  </mh-note>
  …
</mh-canvas>
<script src="mh-canvas.js"></script>   <!-- drag: writes only --x/--y -->

The honest finding

Because position is a data hook, the drag script (mh-canvas.js) only ever writes two numbers — never a style string, never a class. That's the cleanest the hatch can be. But it's still a hatch: connectors don't re-route when you drag (that would need per-note wiring + geometry recompute, fully off-rail), and free positioning can't be expressed as vocabulary. This is the experiment's first real wall — see experiments/saturation.md. MaxHTML's converged, flow-based vocabulary covers document-shaped apps; a spatial canvas is where you eject to a real canvas library.
